# 2026-04-26 TIL

## 오늘 한 일
- Day1에서 만든 기능 중심 코드를 Day2 목표에 맞게 도메인 객체와 서비스 구조로 재배치했다.
- `Main`에서 직접 계산하던 흐름을 `ReservationService.reserve(...)` 호출 중심으로 바꿨다.
- `Room`, `Reservation`, `RoomType`, `MembershipGrade`, `ReservationStatus`를 기준으로 예약 도메인 구조를 정리했다.
- `ReservationValidator`, `PriceCalculator`, `TimeSlotGenerator`가 각각 검증, 가격 계산, 슬롯 생성 책임을 가지도록 역할을 나눴다.

## 오늘 배운 점
- 돌아가는 코드와 설계된 코드는 다르다.
- Day2에서 중요한 것은 기능 추가보다 책임 분리와 객체 협력 구조를 만드는 것이다.
- `main`은 실행 진입점만 담당하고, 실제 흐름은 서비스가 맡아야 읽기 쉬워진다.
- 객체를 만든 뒤 service로 넘기고, service가 validator/calculator에 다시 일을 위임하는 흐름이 객체지향 설계의 시작점이라는 걸 배웠다.
- enum을 쓰면 문자열 비교보다 훨씬 안전하게 상태를 표현할 수 있다.

## 막혔던 부분
- 생성자를 직접 만들고 `new`로 객체를 생성하는 흐름이 아직 익숙하지 않았다.
- `ReservationService`가 정확히 어떤 역할을 해야 하는지 초반에는 감이 잘 오지 않았다.
- `reserve()` 안의 하드코딩 값을 바깥에서 만든 객체와 매개변수로 바꾸는 과정에서 많이 막혔다.
- getter를 통해 객체 값을 꺼내 기존 로직과 연결하는 중간 단계가 특히 어려웠다.

## 해결한 부분
- `Room room = new Room(...)` 형태로 객체를 생성하는 흐름을 이해했다.
- `Reservation reservation = new Reservation(...)` 형태로 예약 정보를 하나의 객체로 묶었다.
- `reservationService.reserve(room, reservation, MembershipGrade.NORMAL)`처럼 main에서 service로 요청을 넘기도록 구조를 바꿨다.
- `ReservationValidator`가 `ReservationStatus`를 반환하도록 구조를 맞췄다.

## 아직 남아 있는 보완점
- `ReservationValidator`의 검증 로직은 지금은 학습용 임시값이 섞여 있으니 다음 단계에서 더 일반화할 수 있다.
- 가격 계산에서 사용 시간, 주말 여부를 예약 정보에서 계산하도록 개선할 여지가 있다.
- 시간 타입은 현재 문자열 기반이므로 이후 `LocalTime` 또는 `LocalDateTime`으로 확장할 수 있다.

## 오늘 피드백
